# Source to Doc

Generate architecture documentation from source code repositories as a Markdown document with Mermaid diagrams.

## Workflow

### Step 1: Gather Context

Before scanning, clarify with the user:
- **Audience**: Engineering team? Leadership? New hires?
- **Depth**: High-level overview or deep technical dive?
- **Focus areas**: Specific systems, services, or modules to prioritize?
- **Exclusions**: Areas to skip (vendor code, generated files, test fixtures)?

Default: engineering audience, moderate depth, full repo coverage.

### Step 2: Scan Repository

Scan in this order to build understanding progressively:

**2a. Project identity**
- Read README.md, package.json, Cargo.toml, go.mod, pyproject.toml, pom.xml (whichever exist)
- Identify: language(s), framework(s), build system, project name

**2b. Directory structure**
- Run `ls` on root, then key subdirectories (src/, lib/, pkg/, app/, services/, cmd/)
- Map the top-level organizational pattern (monorepo, microservices, MVC, hexagonal, etc.)
- Limit tree depth to 3 levels for the document

**2c. Entry points and configuration**
- Find main entry files: main.*, index.*, app.*, server.*
- Read config files: docker-compose.yml, Dockerfile, .env.example, Makefile, CI configs
- Identify deployment targets and infrastructure

**2d. Dependencies**
- Parse dependency manifests (package.json, requirements.txt, go.mod, Cargo.toml, etc.)
- Categorize: runtime deps, dev deps, key frameworks
- Identify external service dependencies (databases, queues, APIs)

**2e. Code patterns**
- Grep for common patterns: routing, middleware, database connections, API endpoints
- Identify architectural patterns: dependency injection, event-driven, layered, etc.
- Look for shared utilities, common interfaces, base classes

**2f. Data flow**
- Trace from entry points through handlers/controllers to data layer
- Identify key data models or schemas (ORM models, protobuf, GraphQL schemas)
- Map inter-service communication if applicable

### Step 3: Analyze and Synthesize

Before writing, organize findings into:
- Component inventory (what exists)
- Relationship map (how components connect)
- Data flow paths (how data moves)
- Technology stack summary

## Mermaid Diagram Rules (Summary)

See [references/mermaid-pdf-guidelines.md](references/mermaid-pdf-guidelines.md) for full rules. Key constraints:

- Max 7-10 nodes per diagram; split into multiple if needed
- Use `graph TD` or `graph LR` only
- Short node labels (max 3 words)
- No nested subgraphs deeper than 1 level
- PDF-safe colors only — no transparency, no light colors that vanish in print
- Every diagram gets its own Markdown heading above it

## Output Format

- Single Markdown file (`.md` extension)
- Save to `docs/` folder in the repo root (create if it doesn't exist, reuse if it does)
- All diagrams as fenced Mermaid code blocks
- Use `##` level headings for major sections (page-break-friendly)
- Keep each section self-contained (readable independently)
- Target 15-30 pages when rendered as PDF
- Never stack diagrams back-to-back; separate with explanatory text

## What NOT to Include

- Line-by-line code explanations
- Auto-generated API reference (reference swagger/openapi instead)
- Test coverage reports or test descriptions
- Git history or changelog
- Performance benchmarks
- Individual file documentation
- TODO/FIXME lists from code
- Installation or setup instructions (belongs in README)
